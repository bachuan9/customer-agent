# 运行与测试手册

## 1. 本地启动

在项目根目录执行：

```powershell
.\.venv311\Scripts\python.exe -m uvicorn app.main:app --reload --app-dir . --port 8001
```

启动成功后访问：

```text
网页工作台：http://127.0.0.1:8001/web
接口文档：http://127.0.0.1:8001/docs
健康检查：http://127.0.0.1:8001/health
```

健康检查正常时返回：

```json
{"status":"ok"}
```

## 2. 环境变量

复制配置模板：

```powershell
Copy-Item .env.example .env
```

默认可以不开启 LLM：

```env
LLM_ENABLED=false
```

如需测试 DeepSeek Function Calling：

```env
LLM_ENABLED=true
DEEPSEEK_API_KEY=你的真实 key
```

注意：不要把 `.env` 或真实 API Key 提交到 GitHub。

## 3. 自动测试

运行全部自动测试：

```powershell
.\.venv311\Scripts\python.exe -m pytest -q
```

也可以只运行 Agent / RAG / LangGraph 主线测试：

```powershell
.\.venv311\Scripts\python.exe -m pytest tests\test_langgraph_agent.py tests\test_langchain_agent.py tests\test_langchain_rag_agent.py tests\test_langchain_tools.py tests\test_langchain_retriever.py tests\test_api_routes.py -q
```

检查前端 JavaScript：

```powershell
node --check web\app.js
```

检查 Python 文件是否能编译：

```powershell
.\.venv311\Scripts\python.exe -m compileall app
```

当前完整检查应看到：

```text
pytest 全部通过
node --check web/app.js 无输出，表示语法通过
python -m compileall app 正常结束
```

如果本地 `.env` 开启了真实 LLM，`pytest -q` 可能因为网络请求、模型超时而变慢。此时可以先运行主线核心测试：

```powershell
.\.venv311\Scripts\python.exe -m pytest tests\test_langgraph_agent.py tests\test_langchain_agent.py tests\test_langchain_rag_agent.py tests\test_langchain_tools.py tests\test_langchain_retriever.py tests\test_api_routes.py -q
```

当前主线核心测试应看到：

```text
94 passed
```

## 5. 手动测试顺序

```text
1. 打开 http://127.0.0.1:8001/web
2. 输入：查订单 A101
3. 再输入：那物流呢
4. 确认 Agent 能根据 A101 自动查到 L101
5. 输入：物流超时政策
6. 确认 Trace 里 RAG 是否命中为“是”
7. 点击“运行Agent评测”，确认 4/4 通过
8. 在 LangGraph 多工具 Agent 实验区点击“订单超时”
9. 确认节点流水线显示：检查待确认 -> 选择工具 -> 调用工具 -> 风险判断 -> 等待确认
10. 确认工具显示为“订单组合工具”，trace 里 tool_selected 为 handle_order_issue
11. 再点击“确认创建投诉”
12. 确认节点流水线显示：检查待确认 -> 创建投诉，并返回投诉编号
```

## 6. LangGraph 实验区测试

LangGraph 实验区用于测试更清晰的多工具 Agent 工作流。

推荐测试输入：

```text
我的订单 A101 48小时了，怎么还没发货
我的物流 L101 48小时了，怎么还没更新
退款多久到账
确认创建投诉
```

期望现象：

```text
订单号 A101：选择 handle_order_issue，展示订单组合工具。
物流号 L101：选择 handle_logistics_issue，展示物流组合工具。
政策问题：选择 search_knowledge，走 RAG 知识库。
高风险问题：先进入 confirm_complaint，不会马上写数据库。
确认创建投诉：进入 create_complaint，写入投诉工单并返回投诉编号。
```

前端重点看：

```text
节点执行流水线
LangGraph Trace
工具结构化结果
风险等级
是否需要确认
投诉编号
```

## 7. 常见问题

### 页面还是旧功能

原因通常是后端没有重启，或浏览器缓存了旧的 `web/app.js`。

处理方式：

```text
重启后端
刷新浏览器
必要时强制刷新 Ctrl + F5
```

如果刚更新过 `web/app.js`，还要确认 `web/index.html` 里的脚本版本号已经变化，例如：

```html
<script src="./app.js?v=20260521-langgraph-flow-v1"></script>
```

### 接口返回 404

常见原因是后端还在跑旧代码。重启后端即可。

### LLM did not return a tool call

意思是大模型没有按 Function Calling 格式返回工具调用。系统会自动降级到规则 Agent，所以这不是项目崩溃。

### LLM request timed out

意思是大模型接口超时。系统会记录降级原因，并退回模板回复。

### Agent Evaluation 为什么默认关闭 LLM

因为自动评测需要稳定可重复，而真实 LLM 可能受网络、超时、模型返回格式影响。Agent Evaluation 主要测试确定性的意图识别、工具调用、RAG、多轮上下文和组合工具链路。
