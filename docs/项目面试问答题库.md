# 项目面试问答题库

这份文档用于准备“电商智能客服 Agent 工作台”项目面试。

学习建议：

```text
先背主链路
再理解每个模块职责
最后用自己的话讲技术取舍
```

项目主链路：

```text
web/app.js
-> /chat
-> routes.py
-> run_agent_with_steps
-> run_agent_trace
-> 意图识别 / LLM 工具选择 / 规则兜底
-> tool_registry.py
-> tools.py
-> db.py
-> reply / steps / trace
-> web/app.js 渲染到客服窗口
```

## 1. 项目整体

### Q1：你这个智能客服 Agent 项目主要解决什么问题？

参考答案：

```text
这个项目模拟电商客服工作台，解决客服在订单查询、物流查询、投诉处理和政策问答中的重复工作。

用户可以在前端输入自然语言，比如“我的订单 A101 48 小时了，怎么还没发货”。
系统会识别用户意图，调用订单、物流、投诉或知识库工具，最后返回客服回复，并展示工具调用 trace。

它不是单纯聊天机器人，而是一个带工具调用、RAG、权限控制、审计日志、Agent Trace 和 Docker 部署能力的后端项目。
```

### Q2：这个项目和普通 CRUD 后台有什么区别？

参考答案：

```text
普通 CRUD 后台一般是用户点按钮，然后固定调用某个接口。

这个项目多了一层 Agent。
用户可以用自然语言表达需求，Agent 负责理解用户意图、选择工具、调用数据库或知识库，再组织回复。

例如普通系统可能需要点“订单查询”按钮并输入订单号。
我的项目里用户可以直接说“查一下 A101 怎么还没发货”，Agent 会自动识别 A101 是订单号，并调用订单/物流组合工具。
```

### Q3：用户从前端输入一句话，到最终返回回复，中间完整链路是什么？

参考答案：

```text
用户在 web/app.js 的客服窗口输入消息。
前端通过 fetch 请求 POST /chat。
FastAPI 的 routes.py 接收 ChatRequest。
routes.py 调用 run_agent_with_steps(req)。
run_agent_with_steps 内部调用 run_agent_trace(req)。
run_agent_trace 会读取 message、user_id、role 和 session memory。
然后系统判断意图，必要时调用 LLM 选择工具，失败时走规则 Agent。
确定工具后通过 tool_registry.py 调用 tools.py 里的真实工具。
tools.py 再调用 db.py 查询或修改 SQLite。
最后 agent.py 生成 reply、steps、trace。
routes.py 返回给前端，web/app.js 把结果渲染成聊天气泡和步骤面板。
```

### Q4：你主要负责了哪些部分？

参考答案：

```text
我主要负责后端 Agent 链路、工具注册与调用、SQLite 数据模型、RAG 知识库、LangChain/LangGraph 实验区、权限和审计日志、前端客服工作台，以及 Docker 部署准备。

更具体地说：
routes.py 负责 API 入口。
agent.py 负责主客服 Agent 的流程。
tool_registry.py 负责工具注册和统一调用。
tools.py 负责订单、物流、投诉、知识库等业务工具。
db.py 负责数据库表和 SQL 操作。
web/app.js 负责前端交互。
```

### Q5：你觉得这个项目最有技术含量的地方是什么？

参考答案：

```text
我认为最有技术含量的是把自然语言客服场景拆成稳定的工程链路。

不是让大模型直接回答一切，而是把订单查询、物流查询、投诉处理、知识库检索封装成工具。
Agent 负责选择工具，工具负责确定性地操作数据库。
同时系统有规则兜底、RAG 命中依据、Trace 展示和高风险操作确认。

这样既能使用 LLM 的自然语言能力，也能保证业务操作可控、可观察、可测试。
```

## 2. Agent 核心

### Q6：客服窗口调用的是哪个 Agent？

参考答案：

```text
客服窗口调用的是主客服 Agent，不是 LangGraph Agent。

实际链路是：
web/app.js
-> POST /chat
-> routes.py 的 chat(...)
-> run_agent_with_steps(req)
-> run_agent_trace(req)

LangGraph Agent 是单独的实验区接口：
/langgraph/agent -> run_langgraph_agent(...)
```

### Q7：规则 Agent、LLM Agent、LangChain Agent、LangGraph Agent 分别负责什么？

参考答案：

```text
规则 Agent 是兜底和确定性流程，适合订单号、物流号、确认操作、投诉状态等明确指令。

LLM Agent 负责让大模型根据自然语言选择工具，也就是 Function Calling。

LangChain Agent 是为了学习和演示 LangChain 风格的工具选择、RAG 链路封装。

LangGraph Agent 是为了把复杂流程拆成状态机节点，比如检查待确认动作、选择工具、调用工具、判断风险、等待用户确认、创建投诉。
```

### Q8：你的 Agent 是怎么判断用户意图的？

参考答案：

```text
主客服 Agent 会先处理一些高优先级意图，比如确认执行、确认创建投诉、取消创建投诉。

然后会识别订单号、物流号、投诉编号、状态更新等明确业务信号。
如果开启 LLM，系统会尝试让 LLM 选择工具。
如果 LLM 没有返回工具、调用失败，系统会降级到规则 Agent。

这样设计的原因是：确定性强的内容用规则更稳定；语义模糊的内容可以交给 LLM。
```

### Q9：如果 LLM 没有返回工具调用，你的系统怎么处理？

参考答案：

```text
系统不会直接报错给用户，而是降级到规则 Agent。

在 trace 里会记录 LLM 降级原因，比如：
LLM did not return a tool call

然后规则 Agent 会继续根据关键词、编号、意图去调用对应工具或知识库。

这样保证即使 LLM 不稳定，客服系统也不会完全不可用。
```

### Q10：为什么要做二次确认？哪些操作属于高风险操作？

参考答案：

```text
二次确认是为了防止 Agent 自动执行高风险写操作。

比如创建投诉、更新订单状态、更新物流状态、修改投诉状态、确认 LLM 建议动作，这些都会影响数据库。

所以系统会先保存 pending action，并提示用户确认。
用户回复“确认执行”或“确认创建投诉”后，系统才真正写数据库。

这属于 human-in-the-loop，意思是关键业务动作必须有人确认。
```

## 3. Function Calling 和工具调用

### Q11：什么是 Function Calling？

参考答案：

```text
Function Calling 是让大模型不要只生成文字，而是根据用户问题选择一个预先定义好的工具，并生成这个工具需要的参数。

比如用户说“查订单 A101”，大模型应该选择 get_order 工具，并给出参数：
order_no = A101

然后后端 Python 代码真正执行这个工具，查询数据库，再把结果组织成回复。
```

### Q12：大模型为什么不能直接改数据库？

参考答案：

```text
因为大模型生成的是文本，它不应该直接操作真实业务数据。

数据库操作必须由后端代码控制。
后端可以做参数校验、权限判断、状态流转校验、审计日志和异常处理。

所以 LLM 只负责“建议调用哪个工具、传什么参数”。
真正执行工具的是 Python 后端。
```

### Q13：tool_registry.py 是干什么的？

参考答案：

```text
tool_registry.py 是工具注册中心。

它有两个作用：
第一，告诉 LLM 系统有哪些工具、每个工具需要什么参数。
第二，根据工具名调用真实 Python 函数。

这样 agent.py 不需要直接知道每个工具的实现细节，只需要把 tool_name 和 arguments 交给 call_tool(...)。
```

### Q14：工具参数从哪里来？

参考答案：

```text
工具参数主要有两个来源。

第一种是规则解析。
比如系统从用户消息里识别出 A101，就把它作为 order_no。

第二种是 LLM Function Calling。
LLM 根据工具描述生成 arguments，比如：
{"order_no": "A101", "query": "我的订单 A101 48小时了怎么还没发货"}

最后后端会把这些参数传给 tool_registry.py，再调用 tools.py 里的具体函数。
```

### Q15：如果工具调用失败，用户会看到什么？

参考答案：

```text
工具调用失败时，系统会返回结构化错误，而不是让程序崩溃。

比如订单不存在，工具会返回 found=false 或 error。
Agent 会根据这个结果组织回复，比如“没有找到该订单，请确认订单号是否正确”。

同时 trace 或工具日志里会记录工具名、参数、是否成功、错误原因，方便排查。
```

## 4. RAG 和知识库

### Q16：你项目里的 RAG 是怎么实现的？

参考答案：

```text
RAG 就是先检索知识库，再基于检索结果生成回复。

在我的项目里，知识来源包括 Markdown 文件和数据库知识。
系统会把知识内容切成 chunk，计算 embedding，并结合关键词分数做混合检索。

用户问政策类问题时，系统先 search_knowledge，找到相关 chunk。
如果启用了 LLM，就把用户问题和命中的知识交给 LLM 生成自然回复。
如果 LLM 不可用，就使用模板回复。
```

### Q17：什么是 chunk？你的知识库是怎么切 chunk 的？

参考答案：

```text
chunk 就是知识库的一小段内容。

不能把整篇文档一次性塞给模型或检索器，所以要切成更小的段落。
比如一篇物流政策文档，可以按标题和段落切成多个 chunk：
退款时效
物流超时
签收异常

这样用户问“物流 48 小时没更新”时，系统更容易命中“物流超时”这一小段，而不是整篇大文档。
```

### Q18：什么是 embedding？你项目里 embedding 起什么作用？

参考答案：

```text
embedding 是把文本转换成一组数字向量。

它的作用是让机器可以比较两段文本的语义相似度。

比如“物流48小时没更新”和“物流超时处理规则”字面不完全一样，但语义接近。
embedding 可以帮助系统发现这种相似关系。

我的项目里用本地教学版 embedding，用于学习和演示 RAG 检索链路。
```

### Q19：如果知识库没有命中，Agent 会怎么回复？

参考答案：

```text
如果 RAG 没有命中，工具会返回 found=false。

Agent 不会编造政策，而是提示暂未找到相关知识，建议客服进一步核实。

这样设计是为了减少幻觉。
对于客服系统来说，不知道就说不知道，比编一个错误政策更安全。
```

### Q20：RAG 和普通关键词搜索有什么区别？

参考答案：

```text
关键词搜索主要看字面有没有匹配。
RAG 检索可以结合 embedding，看语义是否接近。

比如用户说“怎么还没发货”，知识库写的是“物流超时处理”，关键词可能不完全一致。
embedding 能补充语义相似度。

我的项目里使用关键词 + embedding 的混合检索，让结果更稳定。
```

## 5. LangChain 和 LangGraph

### Q21：你为什么引入 LangChain？

参考答案：

```text
引入 LangChain 主要是为了学习和演示标准化的 LLM/RAG 链路。

项目原本已经有自定义 RAG 和 Agent。
LangChain 版本可以把检索、prompt 组装、LLM 调用这类流程表达得更规范。

它不是替代整个项目，而是作为项目里一个独立实验接口，用来展示我理解 LangChain 的基本使用方式。
```

### Q22：LangGraph 和普通函数调用有什么区别？

参考答案：

```text
普通函数调用更像一条顺序代码，从上到下执行。

LangGraph 更像状态机。
它把流程拆成节点，每个节点只做一件事，然后通过条件边决定下一步走哪里。

比如我的 LangGraph Agent 会按节点执行：
检查待确认动作
选择工具
调用工具
判断风险
等待确认或结束

这种方式更适合复杂 Agent，因为流程更清晰，也更容易展示 trace。
```

### Q23：你的 LangGraph 流程有哪些节点？

参考答案：

```text
主要节点包括：

check_pending_confirmation：检查上一轮是否有待确认动作。
select_tool：根据用户问题选择订单、物流或 RAG 工具。
call_tool：真正调用业务工具。
assess_risk：判断是否属于高风险场景。
confirm_complaint：保存待确认投诉动作，等待用户确认。
create_complaint：用户确认后真正创建投诉。
no_tool_reply：没有合适工具时返回兜底回复。
```

### Q24：为什么多工具 Agent 适合用 LangGraph？

参考答案：

```text
多工具 Agent 不只是一次函数调用。
它可能需要先查订单，再查物流，再查知识库，还要判断风险和等待用户确认。

如果全部写在一个大函数里，代码会越来越难读。

LangGraph 可以把每一步拆成节点，每个节点职责清楚，节点之间通过状态传递数据。
这让流程更容易维护、调试和面试讲解。
```

### Q25：LangGraph Agent 和客服窗口主 Agent 有什么关系？

参考答案：

```text
客服窗口主 Agent 是项目默认对话入口，走 /chat。

LangGraph Agent 是项目里的实验区，走 /langgraph/agent。

它们都可以调用订单、物流、RAG 等能力，但定位不同：
主 Agent 更偏实际客服工作台。
LangGraph Agent 更偏展示复杂多工具流程和状态机编排。
```

## 6. 后端工程

### Q26：FastAPI 在你的项目里负责什么？

参考答案：

```text
FastAPI 负责提供后端接口。

比如 /chat 接收客服对话。
/orders 查询和更新订单。
/logistics 查询和更新物流。
/complaints 处理投诉工单。
/knowledge 管理知识库。
/langgraph/agent 测试 LangGraph Agent。

前端通过 fetch 调用这些接口，后端再进入 Agent、工具和数据库层。
```

### Q27：routes.py、agent.py、tools.py、db.py 各自职责是什么？

参考答案：

```text
routes.py 是接口层，负责接收 HTTP 请求和返回响应。

agent.py 是主客服 Agent 层，负责意图识别、工具选择、规则兜底、回复和 trace。

tool_registry.py 是工具注册层，负责根据工具名统一调用工具。

tools.py 是业务工具层，负责订单、物流、投诉、知识库等具体业务动作。

db.py 是数据层，负责 SQLite 建表、查询、插入、更新、删除。
```

### Q28：为什么要分层，而不是都写在一个文件？

参考答案：

```text
分层可以让每个文件职责更清楚。

如果所有逻辑都写在 routes.py，接口、业务判断、工具调用、SQL 会混在一起，后面很难维护。

现在分层后，前端请求只进 routes.py。
Agent 决策放在 agent.py。
业务工具放在 tools.py。
数据库操作放在 db.py。

这样出问题时也容易定位是哪一层的问题。
```

### Q29：投诉工单状态流转怎么保证合法？

参考答案：

```text
投诉状态不是随便改的，而是有允许状态和允许流转。

比如 pending 可以变成 processing。
processing 可以变成 resolved。
但 resolved 不应该随便退回 pending。

后端会先检查新状态是否合法，再检查旧状态到新状态的流转是否合法。
这样可以避免前端传错参数导致业务数据混乱。
```

### Q30：为什么要用 SQLite？

参考答案：

```text
SQLite 适合学习项目和简历项目的本地演示。

它不需要单独安装数据库服务，代码里可以直接连接一个 db 文件。
这让项目更容易启动和交付。

如果未来做生产环境，可以替换成 PostgreSQL 或 MySQL，但当前项目重点是 Agent、RAG 和后端链路，所以 SQLite 足够。
```

## 7. 权限与安全

### Q31：什么是 RBAC？你项目里怎么做权限控制？

参考答案：

```text
RBAC 是 Role-Based Access Control，基于角色的权限控制。

我的项目里有普通客服 agent 和主管 manager。
普通客服可以处理常规客服问题。
主管可以管理知识库、用户、审计日志等更高权限操作。

后端会根据登录用户的角色判断是否允许执行某个接口或工具。
```

### Q32：为什么知识库管理只有主管能操作？

参考答案：

```text
知识库会影响 Agent 的回答。

如果普通客服随意修改知识库，可能导致系统给出错误政策。

所以新增、修改、删除知识库属于管理动作，只允许主管操作。
这样可以保证客服政策内容更稳定、更可控。
```

### Q33：审计日志记录了什么？

参考答案：

```text
审计日志记录谁在什么时候做了什么管理动作，以及是否成功。

比如新增知识、修改知识、删除知识、修改用户角色、重置密码等。

它的意义是可追踪。
如果系统出现错误配置，可以通过审计日志找到操作来源。
```

### Q34：API Key 有没有提交到 GitHub？你怎么避免泄露？

参考答案：

```text
API Key 不会提交到 GitHub。

项目使用 .env 保存本地密钥，.gitignore 和 .dockerignore 都会排除 .env。

README 和部署手册只说明需要配置哪些环境变量，不写真实 key。

Docker compose 运行时通过 env_file 读取本机 .env，而不是把 .env 打包进镜像。
```

### Q35：删除、重置密码、修改角色这类操作怎么防止误操作？

参考答案：

```text
首先，这类操作需要登录和权限校验。

其次，前端会提供明确按钮和确认动作。

后端还会记录审计日志，保存操作者、动作、目标对象和执行结果。

如果未来继续增强，可以再增加二次确认弹窗、操作原因填写和更细粒度权限。
```

## 8. 部署与工程化

### Q36：Docker 在你的项目里有什么用？

参考答案：

```text
Docker 用来把项目和运行环境打包成统一的容器。

项目需要 Python 3.11、requirements.txt 依赖、FastAPI、web 静态文件、SQLite 数据目录和启动命令。

Dockerfile 把这些写成标准镜像。
docker-compose.yml 负责端口、环境变量和数据挂载。

这样别人拉取项目后，不需要手动配置太多环境，就可以用 docker compose up --build 启动。
```

### Q37：Dockerfile 和 docker-compose.yml 分别负责什么？

参考答案：

```text
Dockerfile 负责定义镜像怎么构建。
比如使用 python:3.11-slim，安装 requirements.txt，复制 app、web、docs/knowledge，然后启动 uvicorn。

docker-compose.yml 负责定义容器怎么运行。
比如映射 8001 端口，读取 .env，把 data 和 logs 目录挂载到容器。
```

### Q38：为什么 .env 不应该提交？

参考答案：

```text
.env 里可能有真实 API Key、数据库路径、模型配置等敏感信息。

如果提交到 GitHub，别人可能拿到你的密钥，造成安全风险和费用风险。

所以 .env 应该只存在本地。
项目里可以提交 .env.example，告诉别人需要哪些配置项。
```

### Q39：别人拉你的项目后怎么启动？

参考答案：

```text
本地 Python 方式：
先创建虚拟环境，安装 requirements.txt。
然后运行：
python -m uvicorn app.main:app --reload --app-dir . --port 8001

Docker 方式：
先准备 .env。
然后运行：
docker compose up --build

最后访问：
http://127.0.0.1:8001/web
```

### Q40：本地 Python 启动和 Docker 启动有什么区别？

参考答案：

```text
本地 Python 启动依赖当前电脑的 Python、虚拟环境和依赖安装情况。

Docker 启动把 Python 版本、依赖和启动命令都固定在镜像里。

所以本地启动适合开发调试。
Docker 启动更适合演示、交付和部署。
```

### Q42：怎么确认 Agent 工具调用没有坏？

参考答案：

```text
第一，运行 pytest 自动化测试。
第二，通过前端客服窗口输入典型问题，比如“查订单 A101”“查物流 L101”。
第三，查看 Agent Trace，确认工具名、参数、结果是否符合预期。
第四，查看工具日志，确认工具是否成功执行。
```

### Q43：如果后端接口返回 Not Found，你怎么排查？

参考答案：

```text
我会按链路排查：

先确认后端是否启动成功。
再确认访问端口是否正确，比如 8001。
然后打开 /docs 看接口是否存在。
再检查 routes.py 里有没有注册对应路由。
最后确认前端 fetch 的路径和后端路径是否一致。

之前项目里也遇到过类似问题，很多时候是后端没重启或访问了旧端口。
```

### Q44：如果 LLM 调用失败，系统会不会完全不可用？

参考答案：

```text
不会。

LLM 调用失败时，系统会记录 fallback reason，然后降级到规则 Agent 或模板回复。

订单、物流、投诉这些核心工具仍然可以通过规则执行。
RAG 命中后如果 LLM 润色失败，也可以用模板回复。

这样系统对 LLM 故障更有容错能力。
```

### Q45：你项目现在有哪些不足？

参考答案：

当前项目还是学习和简历展示版本，有几个不足：

embedding 目前是本地教学版，不是真实生产级向量模型。
数据库使用 SQLite，适合演示，但生产环境更适合 PostgreSQL 或 MySQL。
前端是原生 HTML/CSS/JS，后续可以升级 React。
评测数据集还可以扩充，覆盖更多真实客服场景。

## 10. 高频场景追问

### Q46：用户输入“我的订单 A101 48小时了，怎么还没发货”，系统怎么处理？

参考答案：

```text
客服窗口会把消息发到 /chat。
routes.py 调用 run_agent_with_steps。
Agent 识别出 A101 是订单号，并判断这是订单异常场景。
然后调用 handle_order_issue。
这个工具会查订单、关联物流，并检索知识库里的物流超时政策。
工具返回订单状态、物流状态、知识库命中和处理建议。
Agent 根据结果组织客服回复。
前端展示回复和结构化 trace。
```

### Q47：用户第一轮说“查订单 A101”，第二轮说“那物流呢”，系统怎么知道查哪个物流？

参考答案：

```text
这是 session memory 的作用。

第一轮查询订单时，系统会把最近的 order_no=A101 保存到当前用户会话。
第二轮用户说“那物流呢”，虽然没有重新说订单号，但 Agent 可以从 session memory 里取出 A101。
然后根据订单找到关联物流号，再查询物流。

这就是多轮上下文补全。
```

### Q48：为什么订单号和物流号优先用规则识别，而不是完全交给 LLM？

参考答案：

```text
订单号和物流号是非常明确的业务信号。

比如 A101、L101 这种编号，用正则或规则识别更稳定、更快、成本更低。

LLM 更适合处理模糊自然语言。
确定性强的东西交给规则，可以减少误判和 token 成本。
```

### Q49：如果用户说“你好”，为什么不应该创建投诉？

参考答案：

```text
因为“你好”只是普通问候，没有投诉意图，也没有订单、物流、问题描述。

Agent 应该识别为 greeting 或普通对话，并返回欢迎或引导回复。

这类问题说明意图识别要有边界，不能因为客服系统有投诉功能，就把所有输入都当成投诉。
```

### Q50：如果让你继续优化这个项目，你会怎么做？

参考答案：

```text
我会从四个方向优化。

第一，升级真实 embedding 模型和向量数据库，让 RAG 检索更接近生产。
第二，扩展 LangGraph，让主客服窗口也可以逐步接入状态机编排。
第三，增加更完整的 Agent Evaluation 数据集，覆盖更多真实客服场景。
第四，完善部署和监控，比如 Docker 部署、日志追踪、错误告警和接口性能统计。
```

万能回答模板：

```text
这个问题在我的项目里不是直接让大模型处理，而是走一条可控链路：
前端把用户输入发给 FastAPI，
Agent 负责判断意图和选择工具，
工具层执行确定性的业务逻辑，
数据库层保存或查询数据，
最后 Agent 结合工具结果生成回复，并把 trace 返回给前端。

这样设计的好处是可控、可测试、可观察，也方便后续扩展。
```
