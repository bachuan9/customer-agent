# 项目面试讲解

## 1. 项目一句话介绍

这是一个面向电商客服场景的智能客服 Agent 项目，基于 FastAPI、SQLite、原生前端、RAG 知识库和 LLM Function Calling，实现了订单查询、物流查询、投诉处理、知识库检索、多轮上下文、权限控制、工具日志和 Agent 评测。

## 1.1 简历项目描述

项目名称：

```text
电商智能客服 Agent 工作台
```

```text
基于 FastAPI、SQLite、RAG、Function Calling、LangChain 和 LangGraph 实现的电商客服 Agent 系统，支持订单/物流查询、投诉工单处理、知识库检索、多轮上下文、工具调用追踪和高风险操作确认。
```

技术栈：

```text
Python、FastAPI、SQLite、原生 JavaScript、HTML/CSS、RAG、Embedding、LangChain、LangGraph、DeepSeek API、pytest
```

可以放进简历的项目亮点：

```text

2. 接入 DeepSeek Function Calling，由大模型根据用户问题选择工具，并提供规则兜底，提升系统稳定性。

5. 使用 LangGraph 将 Agent 流程拆分为工具选择、工具调用、风险判断、用户确认和创建投诉等节点，实现可观测的多工具工作流。
6. 对订单/物流异常等高风险场景引入 human-in-the-loop 机制，用户确认后才执行创建投诉等写数据库动作。

8. 编写 pytest 自动化测试，覆盖 API、RAG、工具调用、LangChain Agent、LangGraph 工作流和多轮确认闭环。
```

## 2. 核心链路

```text
用户在前端输入问题
-> FastAPI /chat 接收请求
-> agent.py 判断意图
-> llm_agent.py 让大模型选择工具，失败时规则 Agent 兜底
-> tool_registry.py 根据工具名调用真实 Python 工具
-> tools.py 查询订单、物流、投诉或知识库
-> db.py 操作 SQLite
-> llm_reply.py 生成更自然的客服回复
-> routes.py 返回 reply、steps、trace
-> web/app.js 展示回复和结构化 Trace
```

## 3. 可以重点讲的亮点

### LLM Function Calling

项目把订单、物流、投诉、知识库等能力封装成工具，让 LLM 根据用户问题选择工具。对于写操作，系统不会直接执行，而是先保存待确认动作，用户确认后再结合 RBAC 权限执行。

### RAG 知识库

项目支持 Markdown 知识库和数据库知识库，先把知识切成 chunk，再计算 embedding，并结合关键词分数做混合检索。RAG 命中后可以交给 LLM 生成自然客服回复，如果 LLM 失败则退回规则模板。

### Session Memory 多轮上下文

系统会把最近查询到的订单号和物流号保存到 session memory。用户第一轮说“查订单 A101”，第二轮说“那物流呢”，Agent 会自动补全 order_no=A101，并查询关联物流 L101。

### Agent Trace 可观测性

每次回复都会返回结构化 Trace，包括意图、执行模式、工具选择、工具参数、RAG 命中来源、关键词分数、embedding 分数、回复来源和 LLM 降级原因，便于调试 Agent 行为。

### Agent Evaluation 自动评测

项目提供 Agent Evaluation，覆盖单轮订单查询、多轮上下文补全、RAG 政策检索、订单异常多工具处理。评测默认关闭 LLM，保证结果稳定可重复，用于防止 Agent 行为回归。

### LangGraph 多工具工作流

项目额外实现了一个 LangGraph Agent 实验区，用来把 Agent 流程拆成可观察的状态机节点。它不是只做单轮问答，而是把工具选择、工具调用、风险判断、用户确认和数据库写入拆成独立节点：

```text
/langgraph/agent
-> check_pending_confirmation
-> select_tool
-> call_tool
-> assess_risk
-> confirm_complaint 或 create_complaint
```

工具选择策略是：

```text
用户问题包含 A101 这类订单号：调用 handle_order_issue
用户问题包含 L101 这类物流号：调用 handle_logistics_issue
没有明确业务编号：调用 search_knowledge / RAG
```

订单组合工具会同时查订单、关联物流和知识库；物流组合工具会查物流和知识库。工具结果进入风险判断节点，如果命中高风险或工具建议创建投诉，系统不会立即写库，而是保存 pending action，等待用户回复“确认创建投诉”后再写入投诉表。

前端 LangGraph 实验区会展示节点执行流水线、工具名、工具参数、风险等级、确认状态和结构化工具结果，方便调试和演示。

## 4. 面试时可以这样说

```text
这个项目不是简单的聊天机器人，而是一个带工具调用、RAG、上下文记忆和评测体系的客服 Agent。
我把客服能力拆成多个工具，由 LLM Function Calling 或规则 Agent 选择工具；
政策类问题走 RAG 检索，命中知识 chunk 后再生成客服回复；
系统会保存 session memory，实现“查订单 A101”后继续问“那物流呢”的多轮上下文补全；
同时我做了 Agent Trace 和 Evaluation，用来观察和验证 Agent 每次执行是否符合预期。
```

如果面试官重点问 LangGraph，可以这样说：

```text
我后面把 Agent 的一部分流程升级成了 LangGraph 状态机。
比如用户说“我的订单 A101 48 小时了，怎么还没发货”，LangGraph 会先检查是否存在上一轮待确认动作，再根据 A101 识别出订单工具，调用 handle_order_issue 查询订单、关联物流和知识库，然后进入风险评估节点。
如果风险等级是 high，系统不会自动创建投诉，而是进入 confirm_complaint 节点，把 create_complaint 保存成 pending action。
用户下一轮回复“确认创建投诉”后，check_pending_confirmation 节点会识别到这是确认动作，然后跳到 create_complaint 节点，真正写入数据库并返回投诉编号。
这个设计体现了多工具协作、状态机编排、human-in-the-loop 和可观测 trace。
```

## 5. 面试官可能追问

### 为什么订单号和物流号用规则识别，不完全交给 LLM？

```text
订单号、物流号是非常明确的业务信号，用规则识别更稳定、更快、成本更低。
LLM 更适合处理模糊语义和自然语言理解；确定性强的业务编号优先用规则，可以减少误调用和 token 成本。
```

### LangGraph 和普通函数调用有什么区别？

```text
普通函数调用更像一条顺序代码，适合简单流程。
LangGraph 更像一个状态机，每个节点只负责一件事，节点之间通过条件边跳转。
这样复杂 Agent 流程会更清晰，例如先检查 pending，再选择工具，再调用工具，再判断风险，再决定是否确认或结束。
```

### 为什么创建投诉前要用户确认？

```text
创建投诉是写数据库动作，会影响真实业务数据。
所以系统不能因为 Agent 判断高风险就自动建单，而是先保存 pending action，让用户明确确认后再执行。
这属于 human-in-the-loop，能降低误操作风险。
```

## 6. 当前项目边界

```text
当前 embedding provider 是本地教学版 local_hash，适合学习和演示。
LLM 使用 DeepSeek 接口，真实调用可能受网络和模型稳定性影响。
Agent Evaluation 默认关闭 LLM，主要测试确定性的意图、工具、RAG 和上下文链路。
后续可以继续扩展真实向量数据库、更完整的 LangGraph 工作流、生产部署和更完整的评测数据集。
```
